♒︎ Streams: vGitHub (source code):

Design process:

Streams is an experimental product that creates shareable feeds of your work; the prototype linked above organizes GitHub events for any given repo. We iterated on GitHub protoypes over the course of four design sprints, the last two of which were led by me– sketching, prototyping, and user-testing a variety of organizational schemes for your GitHub events. First we decided to simply present a firehose of events, with composable filters being the only way to organize. Based on user-tests, the next iterations experimented with variety of schemas for grouping events. We found that for the most part, people wanted to use this tool to keep track of others' work, not their own. So in the iteration linked above, we chose to group events by branch and pull request– balancing concise high-level overviews of others' work, with the ability to quickly view any details that interest you. If I were to run another design sprint for this prototype, I would add the ability to visualize events across multiple repos and move the filters and title to the side or to a drawer, so that the groups of events become the center of the interface. Our hypothesis was that people would use this to view events (and groups of events), so events should be given visual precedence over the filters and title.

For these experiments, I was entirely responsible for the visual design and its implementation. I also contributed significant amounts of prior user research and application logic (using react + redux).

♒︎ Streams: vLinks

Design process:

This next Streams design sprint was led entirely by me. The goal of this sprint was to design a product for sharing your work with collaborators while substituting the work of app integrations (like GitHub) with manual updates by users themselves. I wanted to find out if all the hassle with different applications' event APIs was worth it or if people would use a separate product that asked them to manually keep track of changes to resources (represented by links). For this sprint I decided to test the figma prototype itself, rather than taking the time to build a working version before we knew if people even wanted it. I conducted four user tests and concluded that the current iteration did not offer enough of an improvement over in-app communication to convince people to use it, so we decided not to build it and saved ourselves significant development time.

Techno-Visual Performance (and server)

Tech used: NodeJS/Express, websockets, Unity, GLSL

I was invited to perform at an audiovisual festival on campus, along with friends who make claymations and techno music. For the show, I developed a NodeJS web app that used websockets to allow people to control 'dancing' characters in the visuals. For the visuals themselves, I created a Unity app that served as the glue between a variety of visual effects:

1) the dancing characters (controlled by viewers' phones via websocket)

2) live-edited GLSL shaders that morphed in response to incoming audio

3) midi-triggered claymation loops and

4) live midi-controlled Unity camera effects.

Over the course of this project I learned how to use websockets to pass data between apps and organized people with divergent talents (animation, music production, and programming) into a cohesive team that produced a brain-melting performance.
